# Requirement Validation Protocol

TDD 시작 전 요구사항의 충분성을 검증합니다.

## 충분한 요구사항의 기준

### 1. 테스트 케이스 도출 가능 여부

- [ ] 정상 케이스 시나리오가 명확한가?
- [ ] 에러/예외 케이스가 식별 가능한가?
- [ ] 입력/출력이 명시되어 있는가?
- [ ] 경계 조건이 정의되어 있는가?

### 2. 인터페이스 발견 가능 여부

- [ ] 주요 컴포넌트/객체가 식별 가능한가?
- [ ] 의존성 관계가 추론 가능한가?
- [ ] 데이터 흐름이 파악 가능한가?

### 3. 검증 기준 명확성

- [ ] "성공"의 정의가 있는가?
- [ ] 수용 기준(Acceptance Criteria)이 있는가?
- [ ] 비기능적 요구사항이 명시되어 있는가? (성능, 보안 등)

## 검증 프로세스

```
요구사항 수신
     ↓
┌─────────────────┐
│ 충분성 체크리스트│
│     검토        │
└────────┬────────┘
         ↓
    ┌────────────┐
    │ 충분한가?  │
    └────┬───────┘
         │
    ┌────┴────┐
    ↓         ↓
  [Yes]     [No]
    ↓         ↓
 계속진행  명확화요청
```

## 불충분시 명확화 요청

AskUserQuestion을 사용하여 누락된 정보를 요청합니다.

### 요청 패턴

**테스트 케이스 관련:**
- "정상적으로 동작하는 시나리오를 구체적으로 알려주세요"
- "에러가 발생할 수 있는 상황은 어떤 것들이 있나요?"
- "입력 데이터의 형식과 예시를 알려주세요"
- "기대하는 출력 결과를 알려주세요"

**인터페이스 관련:**
- "이 기능이 호출하거나 의존해야 하는 다른 컴포넌트가 있나요?"
- "외부 시스템과의 연동이 필요한가요?"
- "데이터는 어디서 가져오고 어디에 저장하나요?"

**검증 기준 관련:**
- "이 기능이 성공했다고 판단하는 기준은 무엇인가요?"
- "수용 기준(Acceptance Criteria)을 알려주세요"
- "성능이나 보안 관련 요구사항이 있나요?"

### 예시

```markdown
요구사항이 다음 부분에서 불충분합니다:

1. **에러 케이스 미정의**
   - 사용자 입력이 유효하지 않을 때 어떻게 처리해야 하나요?
   - 네트워크 오류 시 재시도가 필요한가요?

2. **출력 형식 불명확**
   - 반환 데이터의 구체적인 형식(JSON 구조 등)을 알려주세요

3. **의존성 불명확**
   - 데이터베이스 연동이 필요한가요?
   - 인증/인가 처리가 필요한가요?
```

## 최소 요구사항

TDD를 시작하기 위한 최소한의 정보:

1. **기능 설명**: 무엇을 해야 하는가
2. **정상 케이스**: 최소 1개의 성공 시나리오
3. **입출력 예시**: 구체적인 데이터 예시
4. **성공 기준**: 어떻게 되면 완료인가

이 4가지가 없으면 TDD를 시작할 수 없습니다.
